## 1. 釐清項目對照表

每項格式為：**待釐清 → AI 建議填答 → 理由**。

### A. 輸入模組

**A1. 「字數」如何計算（中英混排、標點、emoji）？**
- **建議填答：** 統一使用碼點數 `Array.from(str).length`。中文字、英文字母、標點、空白各算 1；emoji 以 1 計（碼點層級）。字數統計顯示與「平台字數上限」判斷共用同一個函式 `countChars()`（見附錄 2）。
- **理由：** 原規格各平台「字數上限」未定義計數單位。中文情境用碼點數最直覺，且與使用者在 FB/IG 看到的「字數」概念一致；`Array.from` 能正確處理 emoji 與代理對，避免 `str.length` 把一個 emoji 算成 2。

**A2. sourceText「無上限」與 `max_tokens`、超長分段風險衝突？**
- **建議填答：** 設**軟上限 5000 字**。超過時於輸入框下方顯示「內容較長，建議精簡或分段後再改編（目前 N 字）」，但**不阻擋送出**；不設硬上限。
- **理由：** 原規格 §2.1 寫「無上限」，§7 又寫「超長內容提示分段處理」，兩者矛盾。軟上限兼顧兩者：不擋人，但提醒使用者過長會影響品質與成本。

**A3. 平台字數上限是硬限制還是目標？AI 超出怎麼辦？**
- **建議填答：** 視為**目標上限**。Prompt 中明確要求嚴格遵守。後端產出後對每個平台做字數檢查，若超過該平台上限的 **110%** 即標記 `overLimit: true`。前端字數統計（如 `245 / 220 字`）以**紅色**呈現超限，但**不自動截斷**內容，改由使用者按「重新產生」重抓。
- **理由：** LLM 無法保證精準字數。自動截斷會切斷句子破壞語意，反而更糟；以視覺提示 + 重新產生機制（原規格已有此按鈕）交給使用者把關，較務實。

**A4. tone 預設值？**
- **建議填答：** 維持原規格 `professional`（專業知識感）。
- **理由：** 原規格 §2.1 已明定，符合目標使用者（顧問/知識型自媒體）的主要需求。

### B. 輸出模組

**B1. Tab 顯示順序（依勾選順序 vs 固定順序）？**
- **建議填答：** **固定順序**：facebook → instagram → linkedin → threads → newsletter。只顯示有勾選的平台。
- **理由：** 固定順序讓使用者每次都在同位置找到同平台，避免依勾選順序排列造成的畫面跳動與認知負擔。

**B2. 預設選中哪個 Tab？**
- **建議填答：** 第一個**有結果**的平台（依 B1 固定順序）。
- **理由：** 確保結果一回來就有內容可看，不會停在空白 Tab。

**B3. 字數統計顯示格式？**
- **建議填答：** `125 / 220 字`（目前字數 / 平台上限）。超限時數字轉紅（對應 A3）。
- **理由：** 同時呈現「現況」與「目標」，使用者一眼判斷是否需重抓。

**B4. 「重新產生（單一平台）」的具體行為？**
- **建議填答：** 呼叫獨立 endpoint `/api/regenerate`，**只帶該平台規格**的單平台請求；`max_tokens` 依該平台計算（附錄 2），`temperature` 提高到 **0.9** 以求與前一版有差異；回傳後**只更新該 Tab**，其他 Tab 內容不動。
- **理由：** 對應原規格驗收「單一平台重新產生不影響其他 Tab」。單平台呼叫更省 token、更快（符合 §3.4「單一版本重新產生 < 5 秒」）；提高 temperature 避免重抓出幾乎一樣的結果。

**B5. 複製的內容範圍？**
- **建議填答：** 複製該平台的**純文字結果**；IG 版本包含其 hashtag 區塊。不含「字數統計」等 UI 文字。沿用原規格「複製成功 → 按鈕變『已複製 ✓』→ 1.8 秒還原」。
- **理由：** 使用者複製後是直接貼到平台發文，UI 雜訊不該混入。

**B6. 整批重新產生時是否清空舊結果？**
- **建議填答：** 開始新一輪整批改編時，**保留舊結果**直到新結果成功回傳才整批替換；若失敗則不覆蓋、維持舊結果並顯示錯誤。
- **理由：** 與原規格 §2.3「API 錯誤 → 不清空已有結果」一致，並消除「先清空 → loading → 填回」造成的閃爍。

### C. 驗證邏輯

**C1. 30 字下限？**
- **建議填答：** 維持原規格。空白 → 「請先貼入你想改編的內容」；< 30 字 → 「內容太短，建議至少 30 字以上」。字數以 A1 的 `countChars()` 計。
- **理由：** 原規格 §2.3 已明定，只補上計數定義來源。

**C2. 「至少保留 1 個平台」如何實作？**
- **建議填答：** 當勾選平台只剩最後 1 個時，**鎖定該 checkbox 不可取消**（攔截取消動作並提示「至少需選 1 個平台」）。
- **理由：** 原規格只說「不可全部取消」，未說怎麼擋。鎖定最後一個比「送出時才報錯」體驗更好（即時阻止而非事後罰）。

**C3. 平台字數超限的處理？**
- **建議填答：** 見 A3 —— 標記 + 紅字提示 + 可重新產生，**不阻斷**流程。
- **理由：** 字數超限是品質問題不是錯誤，不該擋住使用者使用其他正常結果。

### D. 技術規格（受架構決策影響最大）

**D1.〔隱性 bug〕`max_tokens: 1000` 固定值在多平台時會截斷 → JSON 解析失敗。**
- **建議填答：** 改為**依勾選平台動態計算**：
  `max_tokens = clamp( Σ(各平台字數上限 × 2.2) + 結構 overhead(256), 下限 1024, 上限 8192 )`
  （2.2 = 中文字 → token 的保守換算係數）。完整公式與範例見附錄 2。
- **理由：** 原規格 §3.2 寫死 `max_tokens: 1000`，但全選 5 平台時，僅 newsletter（800 字）一項就可能逼近 1000 token，5 平台合計遠超過 → 回傳 JSON 被中途截斷 → §3.3「JSON 解析失敗」必然觸發。這是會穩定重現的隱性 bug，必須動態化。

**D2. 架構：前端直呼 → 後端 proxy。**
- **建議填答：** 所有 Anthropic 呼叫移至後端 Route Handler：
  - `src/app/api/repurpose/route.ts`（整批改編）
  - `src/app/api/regenerate/route.ts`（單平台重抓）
  前端只 `fetch` 自家 `/api/*`，永不接觸金鑰。後端用 `@anthropic-ai/sdk`。
- **理由：** common-tech 情境 1/2；消除原規格 §7 最高風險「API Key 前端暴露」。

**D3. SDK 初始化。**
- **建議填答：** `new Anthropic({ apiKey, authToken: null })`；`apiKey` 由後端環境變數讀取（情境 3 的 `getApiKey` fallback helper）。
- **理由：** common-tech 情境 2 ⚠️：不傳 `authToken: null` 會報「Could not resolve authentication method」。

**D4. `anthropic-version` / browser header？**
- **建議填答：** 採後端 SDK 後**不需要** `anthropic-dangerous-direct-browser-access` header（那是前端直呼才需要）；`anthropic-version` 由 SDK 內建帶上。原規格的前端直呼方案**作廢**。
- **理由：** 記錄此差異，避免日後誤加 browser header 或漏設 version。

**D5. system vs user prompt 分工？**
- **建議填答：**
  - **system prompt**：角色設定（內容改編助手）、輸出 JSON 硬規則（只回 JSON、key 為平台代碼、不得有多餘文字）、禁止 markdown 標題 `##` 與粗體 `**`（但**允許** IG 的 `#hashtag`）。
  - **user prompt**：語氣風格描述、各勾選平台規格列表、原始內容注入。
  完整範本見附錄 1。
- **理由：** 把「永遠不變的格式規則」放 system，「依輸入變動的內容」放 user，符合 Anthropic 最佳實務，也讓 prompt 組裝邏輯更清楚。

**D6. temperature？**
- **建議填答：** 首次整批改編 `0.7`；單平台「重新產生」`0.9`。
- **理由：** 首次要穩定可用；重抓時使用者期待「換一個版本」，需更高隨機性。

**D7. 輸出清洗（與「IG 要 hashtag」衝突）？**
- **建議填答：** **per-platform sanitize**（見附錄 2 規則表）：移除 `**粗體**`、`## 標題`、多餘的 ```` ``` ```` 程式碼框與前後空白；**保留** IG 的 `#hashtag` 與段落必要換行。JSON `JSON.parse` 前先跑 common-tech 情境 2 的 `escapeControlCharsInJsonStrings()`。
- **理由：** common-tech 情境 2 建議「乾淨輸出 + regex 兜底」，但本工具 IG 平台明確需要 hashtag，不能一律剝除 `#`，故改成分平台規則。

**D8. JSON 解析容錯？**
- **建議填答：** 解析前先 strip ```` ```json ... ``` ```` 包裹；`escapeControlCharsInJsonStrings()` 後再 `JSON.parse`；若仍失敗，**自動重試 1 次**（重新呼叫一次 API），再失敗才回錯誤碼 `PARSE_ERROR`。
- **理由：** 對應原規格 §3.3「AI 回傳非標準格式」，補上「重試一次」降低偶發失敗對使用者的影響。

**D9. 回傳 key 與勾選對齊？**
- **建議填答：** Prompt 動態組裝時**只要求勾選平台**的 key。後端驗證回傳是否每個勾選平台都有 key；個別平台缺漏 → **只把該平台標記為失敗**（該 Tab 顯示「此平台產生失敗，請重新產生」），**不讓整批失敗**。
- **理由：** 提升韌性：一個平台出問題不該毀掉其他已成功的結果。

**D10. IP 限流（原規格未提）。**
- **建議填答：** 採 common-tech 情境 6：每 IP 每日上限**建議 20 次整批改編**（單平台重抓計較輕或合併計算，可後續調整），用 Upstash Redis（情境 5）、**fail-open**，超量回 HTTP 429 + 友善訊息（對應 D12 錯誤碼）。前端載入時打一支 status endpoint 顯示剩餘次數。
- **理由：** 後端金鑰模式下，成本由 Wing 承擔，必須有用量上限防濫用。

**D11. 帳單支出上限。**
- **建議填答：** 採 common-tech 情境 7：在 console.anthropic.com → Billing → Spend limits 設月支出上限 + 預警 email。列為**上線前必做**。
- **理由：** 限流可能被同 IP/VPN 等繞過，帳單封頂是最後一道防線。

**D12. 友善錯誤系統。**
- **建議填答：** 升級原規格 §3.3 錯誤矩陣為 common-tech 情境 9 的**集中錯誤碼目錄 + 參考編號 `ref`**（`src/lib/error-codes.ts`）。前端 banner 依類別配色：**黃=額度（429）/ 藍=使用者可修（4xx 輸入）/ 紅=系統故障（5xx、timeout、parse）**，附「複製錯誤碼 + ref」按鈕。錯誤碼對照見下表。

| 錯誤碼 | 觸發 | 友善訊息 | 類別/色 | HTTP |
|--------|------|---------|--------|------|
| `INPUT_TOO_SHORT` | < 30 字 | 內容太短，建議至少 30 字以上 | 藍 | 400 |
| `NO_PLATFORM` | 無平台勾選 | 請至少選 1 個平台 | 藍 | 400 |
| `AUTH` | 金鑰錯誤/失效（401/403） | 服務設定異常，請聯絡管理者 | 紅 | 500 |
| `RATE_LIMIT` | 超過每日次數（429） | 今日改編次數已用完，請明天再試 | 黃 | 429 |
| `OVERLOADED` | API 5xx/429/529 重試後仍失敗 | 服務暫時忙線，請稍後再試 | 紅 | 503 |
| `TIMEOUT` | 逾時（>25s） | 產生時間過長，請再試一次 | 紅 | 504 |
| `PARSE_ERROR` | JSON 解析重試後仍失敗 | AI 回覆格式異常，請再試一次 | 紅 | 502 |
| `NETWORK` | 前端 fetch 失敗 | 網路連線異常，請確認後重試 | 紅 | — |

- **理由：** 原規格 §3.3 已有矩陣雛形，升級為帶 `ref` 的集中目錄後，使用者回報 ref 即可 grep 定位 log（情境 9）。註：原「金鑰錯誤」訊息不再叫使用者「確認金鑰」，因金鑰已不在使用者手上。

**D13. 重試 / 退避？**
- **建議填答：** 後端遇 429 / 503 / 529 時做**指數退避重試 1–2 次**，仍失敗才回 `OVERLOADED`。
- **理由：** common-tech 情境 9 的 OVERLOADED 映射；降低瞬時忙線造成的使用者可見錯誤。

**D14. 逾時？**
- **建議填答：** 後端 SDK 呼叫設**硬逾時 25 秒**，逾時回 `TIMEOUT`。對齊原規格 §3.4 效能目標（整批 < 8 秒、單平台 < 5 秒）。
- **理由：** Vercel 免費 serverless function 有執行時間上限（情境 4 ⚠️ ~10–60s），25s 留安全邊際同時不讓使用者無限等待。

**D15. 模型 id？**
- **建議填答：** `claude-sonnet-4-6`，維持原規格、與 common-tech 一致。
- **理由：** 速度/品質平衡，符合本工具需求。

### E. 介面 / 文案（AI 直接草擬，開發時可微調）

**E1. Hero 標題與說明文案。**
- **建議填答：**
  - 標題：**一篇內容，貼一次，產出所有平台的最佳版本**
  - 說明：把你寫好的內容貼進來，選好平台與語氣，AI 會依各平台規則自動改寫字數、語氣與格式，省下每次手動改稿的 30 分鐘。

**E2. Loading 說明文字（多階段輪播）。**
- **建議填答：**「正在讀懂你的內容…」→「為每個平台量身改寫…」→「調整語氣與字數…」→「快好了，正在收尾…」（每 ~2 秒切換）。

**E3. Tips 使用技巧欄（3–5 條）。**
- **建議填答：**
  1. 原文寫得越完整、有重點，改編品質越好。
  2. 同一篇可換不同語氣多試幾次，挑最順的版本。
  3. 不滿意某個平台版本？用「重新產生」單獨重抓，不影響其他平台。
  4. IG 的 hashtag 可依你的受眾再手動微調。
  5. 字數顯示紅色代表超過該平台上限，建議重新產生。

**E4. Header「工具 01/10」是否導覽其他工具？**
- **建議填答：** MVP 範圍**純標示**，不做跨工具導覽（保留未來擴充）。

**E5. 字型載入與 < 1.5s 載入目標。**
- **建議填答：** Noto Sans TC 經 Google Fonts 載入，設 `font-display: swap`，並以系統字（`-apple-system, "PingFang TC", "Microsoft JhengHei", sans-serif`）為 fallback，確保首屏不被字型阻塞。
- **理由：** 對應原規格 §3.4「頁面初始載入 < 1.5 秒」。

**E6. RWD 斷點。**
- **建議填答：** Tailwind `sm`/`md`：手機單欄、桌機卡片排版；結果 Tab 列在手機可橫向捲動。
- **理由：** 對應原規格體驗驗收「行動裝置可正常使用（RWD）」。

**E7. 空狀態。**
- **建議填答：** 首次尚未改編時，結果卡片顯示引導文字「改編結果會出現在這裡 —— 貼入內容、選好平台後按『開始改編』」。

**E8. 瀏覽器支援。**
- **建議填答：** 原規格列 Chrome/Safari，補上 **Edge / Firefox 現代版**。複製優先用 `navigator.clipboard.writeText`，不支援時 fallback `document.execCommand('copy')`。

**E9. 無障礙（原規格未提）。**
- **建議填答：** 基本 `aria-label`、Tab 可鍵盤操作、按鈕焦點樣式、文字/背景對比達 WCAG AA。

## 附錄 2：關鍵技術參數

### 動態 `max_tokens` 公式

```ts
const PLATFORM_LIMIT: Record<string, number> = {
  facebook: 500, instagram: 220, linkedin: 600, threads: 300, newsletter: 800,
};
const CHAR_TO_TOKEN = 2.2;   // 中文字 → token 保守係數
const JSON_OVERHEAD = 256;   // key、引號、括號等結構開銷

function calcMaxTokens(selected: string[]): number {
  const sum = selected.reduce((n, p) => n + (PLATFORM_LIMIT[p] ?? 0), 0);
  const raw = Math.ceil(sum * CHAR_TO_TOKEN) + JSON_OVERHEAD;
  return Math.min(8192, Math.max(1024, raw));
}
```

**範例：**
- 選 3 平台（fb+ig+linkedin = 500+220+600 = 1320 字）→ `1320×2.2+256 ≈ 3160` → `max_tokens = 3160`。
- 全選 5 平台（合計 2420 字）→ `2420×2.2+256 ≈ 5580` → `max_tokens = 5580`。
- 單平台重抓 newsletter（800 字）→ `800×2.2+256 ≈ 2016` → `max_tokens = 2016`。

> 原規格寫死 1000，連 3 平台都不夠 → 截斷 → JSON 解析失敗。動態化後問題消失。

### 統一字數函式

```ts
function countChars(str: string): number {
  return Array.from(str.trim()).length; // 以碼點計，正確處理 emoji / 代理對
}
```

### per-platform sanitize 規則表

| 規則 | 全平台 | 例外 |
|------|--------|------|
| 移除 `**粗體**`、`__底線__` | ✓ | — |
| 移除 `#`/`##` 開頭的 Markdown 標題 | ✓ | **IG 例外**：保留 `#hashtag`（行內或文末 # 開頭詞） |
| 移除多餘 ```` ``` ```` 程式碼框 | ✓ | — |
| 壓縮 3 個以上連續換行為 2 個 | ✓ | — |
| 去頭尾空白 | ✓ | — |
| `JSON.parse` 前跑 `escapeControlCharsInJsonStrings()` | ✓（解析階段，非內容清洗） | — |

> IG sanitize 需謹慎：只移除「Markdown 標題語法的 #」，保留「hashtag 的 #」。實作上 hashtag 特徵為 `#` 後緊接非空白字元且多位於文末；標題特徵為 `#`/`##` 後接空白。
